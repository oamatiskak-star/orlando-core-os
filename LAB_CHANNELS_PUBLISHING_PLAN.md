## 7. BACKLOG-HYGIËNE & DEDUPE STRATEGIE

### Probleem
636 + 519 + 635 = 1.790 video-bestanden. Huidige DB schema = mogelijk duplicate-records (`video_id` dubbel in backlog).

### Dedupe plan

#### Fase 1: Audit (vandaag)
```sql
-- Find duplicates in backlog (same file hash / filename)
SELECT channel_id, content_hash, COUNT(*) as count
FROM youtube_content_queue
WHERE scheduled = TRUE AND scheduled_publish_at IS NULL
GROUP BY channel_id, content_hash
HAVING count > 1;
```
**Verwacht:** 50–200 exact duplicates (rendering pipeline bug).

#### Fase 2: Cleanup
- Delete duplicates, keeping 1 original
- Mark deleted records with `status = 'deduplicated'` (audit trail)
- Recalculate title-uniqueness after dedup

#### Fase 3: Prevent future dupes
- Add unique constraint: `(channel_id, content_hash, scheduled_at)` → prevent re-import
- Rendering-worker: add de-duplication check before queue insertion
- Weekly audit report (Supabase function)

### Content-factory fix
**Huidige rendering-setup genereren duplicates? Probleem:**
- Render pipeline = "create 50 frames, export as MP4" → all 50 = identical video (should be 50 different videos)
- OR: Batch re-render without seeding variation (same seed = same output)

**Actie (voor Orlando):**
- Review rendering pipeline seed/variation logic
- Implement content-hash-based detection upstream
- Scale rendering = scale VARIETY, not volume
